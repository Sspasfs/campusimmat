This is a full revert of release {\ttfamily 3.\+5.\+0}.

{\ttfamily 3.\+5.\+0} contained multiple backwards-\/incompatible (BC) breakages that were not supposed to be released in a minor version, and therefore had to be reverted.

Due to unfortunate branch naming issues when migrating from the old {\ttfamily develop} branch, {\ttfamily develop} was renamed {\ttfamily 3.\+5.\+x}, but contained multiple BC breaks that were fundamentally incompatible with the {\ttfamily 3.\+0.\+0} and newer features.

Due to the quick response time, in order to not break further downstream systems that yet have to receive the update, the {\ttfamily 3.\+5.\+0} release was deleted, so that it does not appear in your system if you accidentally run {\ttfamily composer update}.

Thanks to @fabiang for promptly detecting the issue and reporting it at \href{https://github.com/laminas/laminas-servicemanager/issues/59}{\texttt{ https\+://github.\+com/laminas/laminas-\/servicemanager/issues/59}}\hypertarget{md_public_2glpi_2vendor_2laminas_2laminas-servicemanager_2_c_h_a_n_g_e_l_o_g_autotoc_md6347}{}\doxysubsection{\texorpdfstring{3.\+5.\+1}{3.\+5.\+1}}\label{md_public_2glpi_2vendor_2laminas_2laminas-servicemanager_2_c_h_a_n_g_e_l_o_g_autotoc_md6347}

\item {\ttfamily Config\+Interface} has two important changes\+:
\begin{DoxyItemize}
\item {\ttfamily configure\+Service\+Manager()} now {\bfseries{must}} return the updated service manager instance.
\item A new method, {\ttfamily to\+Array()}, was added, to allow pulling the configuration in order to pass to a Service\+Manager or plugin manager\textquotesingle{}s constructor or {\ttfamily configure()} method.
\end{DoxyItemize}
\item Interfaces for {\ttfamily Factory\+Interface}, {\ttfamily Delegator\+Factory\+Interface} and {\ttfamily Abstract\+Factory\+Interface} have changed. All are now directly invokable. This allows a number of performance optimization internally.

\item {\ttfamily Plugin\+Manager} now enforces the need for the main service locator in its constructor.
